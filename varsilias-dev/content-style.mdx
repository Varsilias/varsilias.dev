Docs should explain how a tool behaves and why its design choices matter.

## Voice

- Use active voice.
- Write in second person when giving instructions.
- Keep sentences short.
- Explain tradeoffs directly.
- Prefer concrete examples over broad claims.

## Headings

Use sentence case for headings. Choose headings that help readers scan for a task or concept.

## Code

All code blocks need language identifiers.

```bash
mint broken-links
```

Use inline code for filenames, commands, paths, options, and package names.

## Claims

Do not claim a package behavior is stable unless it has been verified. Mark provisional APIs clearly when docs are ahead of implementation.
