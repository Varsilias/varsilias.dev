Start with the symptom. Then check signal configuration, adapter registration, exporter state, and queue behavior.

## No logs, metrics, or traces appear

Check these items:

1. Confirm the relevant signal is enabled.
2. Confirm `export.enabled` is explicit when using exporters.
3. Confirm signal overrides under `export.signals` also have `enabled: true`.
4. If using a framework, confirm the adapter is registered.
5. Check `lens.getStats()` for exporter failures or dropped items.

## HTTP metrics or traces are missing

Framework adapters are manual. Make sure both the signal and the adapter are enabled.

```ts
metrics: { enabled: true, http: { enabled: true } },
traces: { enabled: true, samplingRate: 1, http: { enabled: true } },
```

Then register the matching metrics and tracing adapters for Express, Fastify, or Hono.

## Too much route noise

Ignore noisy routes:

```ts
metrics: {
  enabled: true,
  http: {
    enabled: true,
    ignoredRoutes: ['/metrics', '/health'],
  },
},
traces: {
  enabled: true,
  samplingRate: 0.25,
  http: {
    enabled: true,
    ignoredRoutes: ['/health'],
  },
},
```

Health checks and scrape endpoints can dominate telemetry if they are not filtered.

## OTLP export fails

Check the collector base URL and network path.

```bash
curl -v http://localhost:4318
```

Remember that Corelens resolves signal endpoints from the base URL:

- `/v1/logs`
- `/v1/metrics`
- `/v1/traces`

If the collector is unavailable, Corelens should not fail customer requests. Fix exporter delivery separately from request handling.

## Telemetry is being dropped

Check queue settings and `lens.getStats()`.

Common causes:

- queue is too small for traffic volume
- exporter is slow or failing
- `fullQueuePolicy` is dropping new or old items
- shutdown timeout is too short

Increase queue limits carefully. A larger queue can absorb more pressure, but it also uses more memory.

## Trace context is missing from logs

Enable log enrichment:

```ts
logs: {
  enabled: true,
  enrichWithTraceContext: true,
}
```

Also confirm the log is written inside an active span.

## Process hangs during shutdown

Check `shutdownTimeoutMs` and make sure shutdown is owned by one path.

Use either:

- your own `SIGINT` and `SIGTERM` handlers that call `lens.shutdown()`
- `lifecycle.handleProcessSignals: true`

Do not use both in the same app.
