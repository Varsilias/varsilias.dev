Benchmarks are useful when they are reproducible and scoped. Corelens benchmark results should be read as local measurements for specific scripts, not universal claims about every workload.

## What benchmarks cover

The Corelens repository includes benchmark scenarios for logger performance and local telemetry behavior.

Benchmark areas include:

- Corelens logger API
- comparisons against Pino and Winston
- local scenarios from the benchmark scripts

## How to interpret results

Use benchmarks to answer narrow questions:

- did this change make a measured path slower?
- how does this logger path behave under the benchmark script?
- is the local scenario stable enough to compare across commits?

Do not use them as a promise that every production workload will behave the same way.

## What to include with benchmark claims

When publishing benchmark numbers, include:

- Node.js version
- operating system
- CPU details
- benchmark command
- sample size or run count
- relevant Corelens configuration

## Production performance guidance

- Use batch mode for network export.
- Keep queues bounded.
- Use sampling for high-traffic trace workloads.
- Avoid high-cardinality metric labels.
- Exclude noisy routes such as `/health` and `/metrics`.
- Watch `lens.getStats()` during rollout.
