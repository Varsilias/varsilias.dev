Corelens configuration should live close to service startup. Keep it in one file so the telemetry boundary is easy to inspect.

## Minimal configuration

```javascript
import { createCorelens } from "@varsilias/corelens";

export const corelens = createCorelens({
  serviceName: "payments-api",
  environment: process.env.NODE_ENV ?? "development",
  exporter: {
    type: "console"
  }
});
```

## Options

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `serviceName` | `string` | required | Stable service identifier shown in traces and metrics |
| `environment` | `string` | `development` | Runtime environment such as `local`, `staging`, or `production` |
| `sampleRate` | `number` | `1` | Fraction of traces to capture from `0` to `1` |
| `exporter.type` | `console` or `otlp` | `console` | Where telemetry records are sent |
| `exporter.endpoint` | `string` | none | OTLP endpoint when using the `otlp` exporter |

## Environment variables

```bash
CORELENS_SERVICE_NAME=payments-api
CORELENS_ENVIRONMENT=production
CORELENS_SAMPLE_RATE=0.25
CORELENS_EXPORTER=otlp
CORELENS_OTLP_ENDPOINT=http://otel-collector:4318
```

Use environment variables for deployment-specific values. Keep service naming and instrumentation choices in code when they are part of the service contract.

## Sampling

Use full sampling in development and lower sampling in high-volume production services. Metrics should continue to report aggregate behavior even when trace sampling is reduced.

<Tip>
  Start with `sampleRate: 1` until your collector path is stable. Reduce sampling only after you know which spans are useful.
</Tip>
